# Production Deployment Guide

## Overview

`setup-production.sh` builds a complete, self-contained Docker deployment of
Turnstone inside this `production/` folder. It uses the released images from
`ghcr.io/turnstonelabs/turnstone` (no local build), fronts the dashboard with
the bundled Caddy service for HTTPS, and generates every file needed so the
stack starts with a single `docker compose up -d` from this folder.

The committed contents of this folder are only:

| File | Purpose |
|------|---------|
| `setup-production.sh` | Interactive deployment builder |
| `deployment.md` | This guide |

Everything else is generated by the script and ignored by Git:

| Generated file | Purpose |
|----------------|---------|
| `compose.yaml` | Production stack (Postgres, console, server, channel, Caddy, SearxNG) |
| `override.compose.yaml` | Deployment-specific adjustments (OIDC wiring, public HTTPS) |
| `tls.compose.yaml` | Service-to-service mTLS overlay (only when TLS is enabled). The script sanitizes the copy: the console `command:` override from the upstream overlay is removed because it carries a `--poll-interval` flag the console CLI no longer accepts (the console falls back to the base compose command). |
| `Caddyfile` | Browser TLS termination for the dashboard |
| `caddy/Dockerfile` | Caddy image with a DNS-challenge plugin (only for public DNS) |
| `config/turnstone-oidc.env` | OIDC settings (only when SSO is enabled) |
| `searxng/` | Bundled SearxNG configuration for the `web_search` tool |
| `.env` | Secrets, image tag, and the compose file chain |
| `setup-production.env` | Persistent answers file — see [Answers file](#answers-file) |

## Prerequisites

- A Linux host with Docker Engine and the Docker Compose v2 plugin
- For a public deployment: a DNS record for the chosen hostname pointing at
  this host, ports 80 and 443 reachable from the internet, and API
  credentials for the DNS provider (Cloudflare, Route 53, or Duck DNS)
- For SSO: a confidential OIDC client registered at the identity provider
  with the redirect URI `https://<turnstone-host>/v1/api/auth/oidc/callback`
- The repository cloned locally:

```bash
git clone https://github.com/turnstonelabs/turnstone.git ~/turnstone
cd ~/turnstone
```

## Running the script

```bash
cd production
./setup-production.sh
```

### Order of operations

1. **Deployment mode** — production or development. Choosing development
   ends the script and directs you to `./run.sh` in the repository root,
   which starts the local development stack.
2. **Image tag** — track the rolling `latest` image, or pin a released tag.
   The pinned path looks up the newest release tag automatically and lets
   you confirm or replace it.
3. **OIDC single sign-on** (yes/no) — see below.
4. **TLS** (yes/no) — enables mutual TLS between the Turnstone services via
   the `tls.compose.yaml` overlay (an init container bootstraps an internal
   CA; services auto-provision certificates through the console's ACME
   endpoint — see `docs/tls.md`).
5. **Public DNS** (yes/no) — serve the dashboard at a public hostname with
   a browser-trusted Let's Encrypt certificate on port 443.
6. **LLM backend** (yes/no) — bootstrap default for an OpenAI-compatible
   backend. Backends can also be connected later in the console Models tab.
7. **Channel gateway** (yes/no) — Discord and/or Slack tokens.
8. **File generation** — the stack files are copied into this folder, the
   overlays and `Caddyfile` are written, secrets (`TURNSTONE_JWT_SECRET`,
   `POSTGRES_PASSWORD`) are preserved from the existing `.env` if present
   (or generated fresh on a first run), and `.env` is written with
   `COMPOSE_FILE` chaining every applicable compose file.

Answering **no** to every optional section still produces a safe and secure
deployment: random secrets, the dashboard served over HTTPS on
`https://localhost:8443` via Caddy's local CA, and no ports published beyond
Caddy and localhost-bound Postgres.

### Information needed per section

**OIDC single sign-on**

- Provider type: `authentik` or `generic`.
- Authentik: base URL, application slug, client ID, client secret, the
  Authentik group mapped to the Turnstone admin role, and the group mapped
  to general user access. The issuer is derived as
  `https://<authentik-host>/application/o/<slug>/` and the role claim is
  `groups` (Authentik's group membership claim).
- Generic: issuer URL, client ID, client secret, login button label,
  scopes, role claim, the claim value mapped to the admin role, the claim
  value mapped to general users, and optional extra trusted endpoint
  hostnames.
- Both: the public origin browsers use to reach Turnstone (the redirect
  base), whether the provider resolves to a private/internal address, and
  whether to disable password logins (SSO-only mode).

Role bindings map onto the built-in roles: the admin value →
`builtin-admin`, the general-users value → `builtin-operator`. See
`docs/oidc.md` for the full reference.

**TLS / public DNS**

- TLS: no further input; the mTLS overlay is added to the compose chain.
- Public DNS: the hostname Turnstone should be reachable at, a Let's
  Encrypt registration email, the DNS challenge provider (Cloudflare,
  Route 53, or Duck DNS), and that provider's credentials:
  - Cloudflare: a DNS API token with `Zone:DNS:Edit` (`CF_DNS_API_TOKEN`)
  - Route 53: an AWS access key ID and secret access key
  - Duck DNS: the account API token
- The script writes a `Caddyfile` for the domain with a DNS-01 challenge
  and a `caddy/Dockerfile` that builds Caddy with the matching DNS plugin,
  publishing ports 80 and 443.

**LLM backend**

- OpenAI-compatible base URL, API key, and an optional default model alias.

**Channel gateway**

- Discord: bot token and guild ID. Slack: bot token and app-level token.

## Answers file

`setup-production.sh` maintains a persistent answers file at
`production/setup-production.env`. The file is written with mode 600 and
is gitignored (it may contain OIDC client secrets and channel tokens).

### What the file does

- **First run** — the file does not exist yet; the script creates it at the
  end of a successful run, recording every answer given.
- **Subsequent runs** — answers from the previous run are loaded as defaults
  before any prompt is shown. If you press Enter at a prompt without typing
  a new value, the prior answer is reused.
- **After each run** — the script writes the current answers back so the
  file stays up-to-date. It **never** modifies the `AUTORUN` flag.

### AUTORUN flag

The first non-comment line of the file is:

```
AUTORUN=false
```

| Value | Behaviour |
|-------|-----------|
| `false` (default) | Script prompts interactively; values in the file are pre-filled defaults. |
| `true` | Fully non-interactive. The script uses only values from the file (and any exported environment variables); it never stops for user input. If a required value for an enabled section is missing the script exits immediately with an error naming the missing variable. |

**Only you should change this flag.** `setup-production.sh` never modifies it.

### Input precedence

From highest to lowest:

1. **Exported shell environment variable** — set before running the script,
   e.g. `TURNSTONE_SETUP_OIDC=n ./setup-production.sh`
2. **Value in `setup-production.env`** — loaded at startup
3. **Interactive prompt / built-in default** — what you type at the prompt,
   or the bracketed default if you press Enter

### Non-interactive runs

**Via the answers file (recommended):**

1. Run the script once interactively to generate and populate
   `setup-production.env`.
2. Edit the file to set `AUTORUN=true`.
3. On future reruns the script completes with zero prompts.

**Via exported environment variables (original method, still works):**

```bash
TURNSTONE_SETUP_MODE=production \
TURNSTONE_SETUP_IMAGE_CHANNEL=latest \
TURNSTONE_SETUP_OIDC=n \
TURNSTONE_SETUP_TLS=n \
TURNSTONE_SETUP_PUBLIC_DNS=n \
TURNSTONE_SETUP_LLM=n \
TURNSTONE_SETUP_CHANNELS=n \
./setup-production.sh
```

Exported variables override both the answers file and interactive prompts.

## Starting the stack

```bash
cd production
docker compose up -d
```

`.env` sets `COMPOSE_FILE`, so plain `docker compose` commands in this folder
automatically pick up every generated overlay.

## Post-deploy

1. Open the dashboard:
   - Public DNS: `https://<your-hostname>`
   - Otherwise: `https://localhost:8443` and trust Caddy's local root once:
     `docker compose exec caddy cat /data/caddy/pki/authorities/local/root.crt`
2. Create the first admin account via the setup wizard (required even in
   SSO-only mode — the wizard is only available while no users exist).
3. Open the **Models** tab and connect at least one LLM backend, e.g. an
   OpenAI-compatible endpoint such as `http://host.docker.internal:8000/v1`.
4. If SSO is enabled, sign in with the identity provider and confirm the
   admin and general-user group bindings behave as expected.

## Maintenance

- Re-run `./setup-production.sh` whenever hostnames, credentials, or
  feature choices change; it regenerates the files in place.
- `.env` and `config/turnstone-oidc.env` contain secrets and are written
  with mode 600 — keep them private.
- `setup-production.env` is also mode 600 and gitignored; it may contain
  OIDC client secrets and channel tokens.
- To update a pinned deployment, re-run the script and pin the new tag,
  then `docker compose up -d` to roll the services.

## Recovering from a failed or interrupted deployment

Both the setup script and the compose stack are safe to re-run. Nothing
about a failed first attempt is permanent — pick the right rerun below.

### Which command to re-run

| Situation | Fix |
|-----------|-----|
| `docker compose up -d` failed or was interrupted (pull error, build error, unhealthy container, network drop, Ctrl-C) | Fix the underlying cause if any, then run `docker compose up -d` again from this folder. It is idempotent: finished steps (pulled images, created volumes/networks, healthy containers) are kept and only the missing pieces are retried. **Do not** re-run the setup script for this. |
| You answered a setup prompt incorrectly (wrong hostname, token, group name, …) | Re-run `./setup-production.sh` with the corrected answers, then `docker compose up -d` to apply. The script regenerates the files in place; existing data volumes are untouched. |
| You want a completely clean retry | `docker compose down` (add `-v` **only** if you also want to delete the database and all data), re-run `./setup-production.sh` if the configuration needs to change, then `docker compose up -d`. |

### Common first-run interruptions and what they mean

- **`pull access denied for turnstone-caddy, repository does not exist`** —
  older generated overrides let Compose try to *pull* the locally-built
  Caddy DNS image before building it. The message was harmless (Compose
  fell back to building the image) but alarming. Overrides generated by the
  current script set `pull_policy: build` so the pull is never attempted;
  re-run `./setup-production.sh` to regenerate `override.compose.yaml` if
  you still see it.
- **`WARN Docker Compose requires buildx plugin to be installed`** — the
  Caddy DNS image was built with Docker's deprecated legacy builder because
  the buildx plugin is missing. The build still works today; install
  `docker-buildx-plugin` to remove the warning and future-proof builds.
- **Network failures while pulling images** (timeouts, DNS errors,
  `connection reset`) — nothing to clean up. Restore connectivity and run
  `docker compose up -d` again; already-downloaded layers are cached and
  the pull resumes.
- **`dependency failed to start: container turnstone-console-1 is
  unhealthy`** (or another service reported unhealthy) — a service did not
  pass its health check within the allowed window, so dependents were not
  started. On a first boot this is often transient (database
  initialisation, mTLS certificate bootstrap, and image extraction all
  land at once). Check the reason first:

  ```bash
  docker compose ps
  docker compose logs console        # or the failing service name
  ```

  If the logs show only slow startup, `docker compose up -d` again — the
  already-healthy services are left alone and only the failed ones are
  restarted. If the logs show a configuration error (for example an
  unreachable OIDC issuer, or an issuer that resolves to a private address
  without the private-network opt-in), re-run `./setup-production.sh` with
  corrected answers and then `docker compose up -d`.
- **Let's Encrypt / DNS-challenge errors in the caddy logs** — verify the
  DNS record points at this host and the DNS provider token is valid, then
  `docker compose up -d` (or `docker compose restart caddy`). Caddy retries
  certificate issuance on its own; no state needs deleting.

### Rerun behaviour of `setup-production.sh`

- The script only writes files; it never starts containers, so stopping it
  midway leaves the running stack (if any) untouched. If it stops before
  the final "Setup complete" message, simply run it again — partially
  generated files are overwritten.
- On rerun, prior prompt answers are loaded from `setup-production.env` and
  used as defaults (or used without prompting when `AUTORUN=true`).
- **`TURNSTONE_JWT_SECRET` and `POSTGRES_PASSWORD` are automatically
  preserved from the existing `.env` on every rerun.** The script reads the
  current values from `.env` before overwriting it, so the Postgres data
  volume and the new `.env` always stay in sync. Fresh random values are
  only generated on a first run when no prior `.env` exists.

## Troubleshooting

### `FATAL: password authentication failed for user "turnstone"`

**Cause:** The `POSTGRES_PASSWORD` in `.env` does not match the password
stored in the Postgres data volume (`turnstone_postgres-data`, or
`${COMPOSE_PROJECT_NAME}_postgres-data` if `COMPOSE_PROJECT_NAME` is set).
This happens when an existing volume is paired with a newly-generated
password — the Postgres image only applies `POSTGRES_PASSWORD` on first
initialisation of an empty data directory and never updates an existing one.

**Common scenario — fresh clone on a host that previously ran Turnstone:**
After `git clone` and `./setup-production.sh`, no `.env` exists yet, so
the script would normally generate a fresh random password. If the Docker
named volume `turnstone_postgres-data` still exists from a prior deployment,
that new password would not match the one stored in the volume.

**How `setup-production.sh` now handles this:**

Before generating a new `POSTGRES_PASSWORD` the script checks whether the
volume `turnstone_postgres-data` (or `${COMPOSE_PROJECT_NAME}_postgres-data`
if that variable is set) already exists:

- **Interactive run (TTY present):** you are warned and offered three choices:
  - **enter** — type the existing password; it is stored in `.env` and the
    volume is reused with all its data intact.
  - **wipe** — the volume is removed (all data deleted) after explicit
    confirmation, and a fresh password is generated.
  - **abort** — the script exits with instructions.

- **Non-interactive run (`AUTORUN=true` or no TTY) without a supplied
  password:** the script exits immediately with an error that names the volume
  and lists the three remediation options. Supply the password in one of these
  ways then rerun:
  1. Export it before running: `export POSTGRES_PASSWORD=<existing-password>`
  2. Add it to `setup-production.env`:
     ```
     POSTGRES_PASSWORD=<existing-password>
     ```
  3. Or remove the stale volume first (destroys all data):
     ```bash
     docker volume rm turnstone_postgres-data
     ```

**If you are already in a broken state** (passwords diverged before this
version of the script), choose one of these remedies:

1. **Restore the old password** — if you have the old `POSTGRES_PASSWORD`
   value, put it back in `production/.env` then run `docker compose up -d`.

2. **Update the database password** — connect to the running Postgres
   container and change it to match the value now in `.env`:

   ```bash
   docker compose up -d postgres
   docker compose exec postgres psql -U turnstone \
       -c "ALTER USER turnstone WITH PASSWORD '<value from .env>';"
   docker compose up -d
   ```

3. **Full wipe** — if you have no data worth keeping, destroy the volume
   and let Postgres reinitialise with the current password:

   ```bash
   docker compose down -v
   docker compose up -d
   ```
